Symmetric Encryption:
same key exchanged between sender and receiver, used to encrypt and decrypt data. Risky method
To safely deliver symetric key from client to server is used Asymetric Encryption.
$ openssl genrsa -out my-bank.key 11024
$ openssl rsa -in my-bank.key -pubout > mybank.pem


Asymetric Encryption:
pair of keys private(-key, .key) and public (pem, crt).

User -         $ ssh-keygen
Web Server -   $ cat ~/.ssh/authorized_keys

$ openssl req -new -key my-bank.key -out my-bank.csr -subj "/C=US/ST=CA/O=myOrg, Inc./CN=my-bank.com"

Certificates have to be signed by Certificate Authority

Root Certificates - used to sign Server Certificates

Certificate Creation for cluster (easyRSA, OPENSSL)
  generation ca.key:
  $ openssl genrsa -out ca.key 2048
  output: ca.key

  certificate Signing Request:
  $ openssl req -new -key  ca.key -subj "/CN=KUBERNETES-CA" -out ca.csr
  output: ca.csr

  Sign Certificates
  $ openssl x509 -req -in ca.csr -signkey ca.key -out ca.crt
  output: ca.crt

Admin User
   generation ca.key:
   $ openssl genrsa -out admin.key 2048
   output: admin.key

   certificate Signing Request:
   $ openssl req -new -key  admin.key -subj "/CN=kube-admin/OU=system:masters" -out admin.csr
   output: admin.csr

   Sign Certificates
   $ openssl x509 -req -in admin.csr -CA ca.crt -CAkey ca.key -out admin.crt
   output: admin.crt

 Same process for: Client Certificates and other components that access the kubeAPI server:
   - kube_scheduler as part of system has to be prefixed by SYSTEM:KUBE_SCHEDULER
   - same with Kube Controller Manager - SYSTEM:KUBE-CONTROLLER-MANAGER
   - kube-proxy (SYSTEM: prefix not needed)

  Next you can provide certificates as parameters:
    $ curl https://kube-apiserver:6443/api/v1/pods \

        --cacert ca.crt

 Server side certificates:

   etcd Server: etcdserver.crt, etcdserver.key, etcdpeer1.crt (in case high availability where etcd is deployed as cluster across multiple servers) Check etcd.yaml:
     -> etcd peer (etcdpeer1.crt, etcdpeer1.key)
     -> etcd peer (etcdpeer2.crt, etcdpeer2.key)  
     
   Kube-api Server: apiserver.crt, apiserver.key (every talk to api server):
     $ openssl genrsa -out apiserver.key 2048
     output: apiserver.key

   Certificate Signing Request:
     $ openssl req -new -key  apiserver.key -subj "/CN=kube-apiserver" -out apiserver.csr -config openssl.cnf
     output: apiserver.csr


   How specify alternate names?
   For that you must create open SSL config file: openssl.cnf and pass this config file as the option while creating apiserver.csr (the above Certificate Signing Request)
   ```
   [req]
   req_extensions = v3_req
   distinguished_name = req_distnguished_name
   [ v3_req ]
   basicConstraints = CA:FLASE
   keyUsage = nonRepudiation,
   subjectAltName = @alt
   [alt_names]
   DNS.1 = kubernetes
   DNS.2 = kubernetes.default
   DNS.3 = kubernetes.default.svc
   DNS.4 = kubernetes.default.svc.local
   IP.1 = 10.96.0.1
   IP.2 = 172.17.0.87
   ```
   Sign Certificates
     $ openssl x509 -req -in apiserver.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out apiserver.crt -extensions v3_req -extfile openssl.cnf -days 1000
     output: apiserver.crt 
   next pass certificates to kubeAPI servers executables: https://kubernetes.io/docs/reference/command-line-tools-reference/kube-apiserver/

   Kubelet server: (kubelet.crt, kubelet.key). Certificates will be named afer their nodes: node01, node02, node03. Once certificates will be create use them in kubelet-config.yaml
   Must be done for each node in the cluster.
```
     kind: kubeletConfiguration
     apiVersion: Kubelet.config.k8s.io/v1beta1
     authentication:
       x509:
         clientCAFile: "/var/lib/kubernetes/ca.pem"
     authorization:
       mode: Webhook
     clusterDomain: 'cluster.local"
     clusterDNS:
       - "10.32.0.10"
     podCIDR: "${POD_CIDR}"
     resolvConf: "/run/systemd/resolve/resolve.conf"
     runtimeRequestTimeout: "15m"
     tlsCertFile: "/var/lib/kubelet/kubelet-node01.crt"
     tlsPrivateKeyFile: "/var/lib/kubelet/kubelet-node01.key"
```
  Kubectl Nodes (Client Certificates) kubelet-client.crt, kubelet-client.key . Certificates will be named as following (node keyword) system:node.node01, system:node.node02, system:node.node03.
  After creating certs, add them to kube-config file.
